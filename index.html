Meal Prep OS v1.0 · What this system is · The weekly workflow · First-time setup order
What is 29 Performance Meal Prep OS?

A single-file kitchen intelligence system for athlete meal prep businesses. Everything — ingredient costs, recipe builds, weekly menus, batch production, staff scheduling, purchasing, and multi-week financial history — lives in one HTML file. No server, no subscription, no internet required. Open it in any modern browser and you have a full operational OS.

All data saves automatically to your browser's localStorage. Export a JSON backup regularly (sidebar → ⬇ Export) to protect your data.

The golden rule: always build bottom-up. Ingredients must exist before recipes, recipes before the menu, and a populated menu before Production / Purchasing / Packaging will have anything to calculate. Labour and Cost Stack work in parallel with the menu.
The Weekly Workflow — Step by Step
1
Build the weekly menu
Go to Weekly Menu. Add each dish, assign recipe components, enter Session 1 (S1) and Session 2 (S2) portion counts for both cooking sessions. Every downstream module (Packaging, Production, Purchasing) pulls from this.
2
Schedule labour
Go to Labour. Add staff shifts for the week. The system calculates total hours, total cost, and — critically — labour cost per portion, which auto-syncs into the Cost Stack.
3
Confirm costs
Go to Cost Stack. Labour per Dish is already populated from your shifts. Verify packaging costs and overheads. Check your gross margin is ≥50%. Adjust sell prices on the menu if needed.
4
Run operations
Use Packaging as the kitchen pack guide (printable). Use Production for batch quantities and station instructions. Use Purchasing for your consolidated supplier order list.
5
Manage deliveries & print labels
In the Delivery module: add customers, create orders allocating dishes per customer, fill delivery confirmations on the Route Sheet, and print Dymo 41×89mm labels from the Labels tab.
6
Archive the week
Click 💾 Archive This Week from the Menu, Labour, or Weekly Archive page. A checklist modal will flag any gaps (missing S2 portions, no shifts scheduled, zero-value cost items) before you confirm. The archive captures a full snapshot: every dish, every shift, and the complete P&L in one record.
First-Time Setup Order
1
Add your ingredients
Go to Ingredients and add every raw ingredient you use. Fill in AP cost per 100g, processing yield %, cook yield %, and all nutrition fields. This is the most important step — every cost and nutrition figure downstream flows from here.
2
Build your recipe components
Go to Recipes and build each component (Protein, Carb, Veg, Condiment, Complete). Quantities are single-serving raw weights in grams. Add cook method, station, timings, and step-by-step instructions.
3
Add your staff roster
Go to Labour and add every team member with their role, hourly rate, and status. Do this once — you'll schedule them week by week using this roster.
4
Set up your cost stack
Go to Cost Stack and update the packaging lines to match your actual box, lid, and sleeve costs. Labour per Dish will auto-fill once you have shifts. Add any overhead or delivery cost lines your business carries.
5
Then follow the weekly workflow above
Once master data is set up, the weekly cycle is: build menu → schedule labour → check costs → run operations → archive.

The Labour → Cost Stack Auto-Sync

When you open the Cost Stack, it reads your current shifts and staff rates, calculates total labour cost ÷ total portions, and updates the Labour per Dish line automatically. This means your cost model always reflects the actual team schedule — not a stale manual figure. The green AUTO badge on the Labour per Dish line confirms it's live-synced. If you need to override it (e.g. for a quote before shifts are scheduled), click ✏ Override to unlock the field, and ↩ Auto to restore sync.

What Updates the Dashboard?

The Dashboard reads live from the current weekly menu every time you visit it. KPIs, alerts, volume chart, and macro profile are all real-time — no cache to clear. If the dashboard looks wrong, the underlying menu or recipe data needs fixing.

Yield % Explained — Worked Examples
Processing Yield (trimming)
Example: Broccoli — 81% processing yield
You buy 1,000g of broccoli.
You trim the stalks and lose 190g.
Usable florets: 810g = 81% of what you paid for.

EP Cost = AP Cost ÷ (Proc% ÷ 100)
If AP = £0.219/100g → EP = £0.219 ÷ 0.81 = £0.270/100g
Cook Yield (moisture loss or absorption)
Example: Chicken Breast — 75% cook yield
You start with 165g raw chicken per serving.
After oven cooking, it loses 25% moisture.
Cooked weight: 165 × 0.75 = 123.75g on the plate.

This is the weight that appears on your Packaging sheet.
Why does rice show 333%? Rice is the exception to the rule. Raw dry rice absorbs water as it cooks, so the cooked weight is significantly heavier than the raw weight. 75g dry basmati + absorbed water ≈ 250g cooked. That's 333% of the raw weight. Enter the cook yield as 333 for rice. The system handles numbers over 100% correctly.
Total Yield
Combined processing + cook yield
Total Yield = (Proc% ÷ 100) × (Cook% ÷ 100)

Broccoli: 0.81 × 0.60 = 0.486 — 130g raw → 63g cooked florets
Rice: 1.00 × 3.33 = 3.33 — 75g raw → 250g cooked rice
Nutrition Fields

All nutrition is entered per 100g raw weight. The system scales values by the quantity used in each recipe line. Core macros (kcal, protein, carbs, fat, fibre, salt) flow into recipe cards and the weekly menu macro display. The extended micronutrient fields (vitamins, minerals) appear in ingredient detail views and the RDA comparison panel.

Use nutrition data from the manufacturer label, Cronometer, or the Open Food Facts database for accuracy. Where exact data isn't available, use a trusted source like the USDA FoodData Central.

Editing and Deleting Ingredients

Click the ✏ pencil icon on any ingredient row to open the edit modal with all current values pre-filled. Click ✕ to delete — note that deleting an ingredient that is used in recipes will cause those recipe lines to show as missing. Always check the Recipe Engine after deleting an ingredient.

The Single-Serving Rule
All recipe quantities are per single serving, in raw grams. The system scales them up to batch size in Production automatically. Never enter batch quantities here — if you change the recipe's serving size, all downstream batch calculations will be wrong.
Duplicating & Versioning Recipes

Use the ⎘ Duplicate button (the copy icon on every recipe row) to open the Recipe Duplicate modal. This is the recommended approach for versioning recipes, creating athlete-specific variants, or trialling macro adjustments without touching your proven baseline.

Modal fieldWhat to do
New Recipe NamePre-filled with the source name. Edit freely — e.g. Chicken Tikka v2 or Tikka Bowl (High Protein)
Variant TagClick a preset (v2, v3, Athlete, High Protein, Low Carb, Bulk, Cut) or type a custom tag. It appends to the name automatically.
StatusDefaults to Draft so the copy stays off the live menu until you are ready to promote it.
Athlete variant example: Duplicate your standard Tikka Bowl → tag it Athlete → increase protein quantity → set to Active when ready. The original stays untouched for your general menu.
Versioning tip: When rolling a recipe update live, keep the original as Draft until the new version has been archived at least once. That way the P&L history for older weeks still reflects the correct recipe cost.
What the Cooked Weight Column Shows

The cooked weight in the recipe list is the sum of all ingredient lines multiplied by their total yield (processing × cook yield). This is the weight that appears on the Packaging sheet — what physically goes in the box. It is not the raw weight and not a target — it is a calculation.

Athlete Scaling Explained

The scaling system is designed for athlete meal prep where different customers need different macro targets. Here's how to use it:

Example: bulking athlete vs cutting athlete, same dish
Base recipe: Chicken Tikka = 165g raw chicken → ~124g cooked

Bulking athlete (Protein Scale = 1.3): 165 × 1.3 = 214.5g raw chicken → ~161g cooked
Cutting athlete (Protein Scale = 0.8): 165 × 0.8 = 132g raw chicken → ~99g cooked

All cost and macro values update automatically based on the scale applied.
Tip: Use Overall Scale for meal plans that are uniformly larger or smaller across all components (e.g. a premium 1.2× plan). Use individual component scales (P / C / V) for macro-specific adjustments.
Session 01 and Session 02

Each dish has two portion fields — Session 01 and Session 02 — representing your two cooking sessions each week. The total of S1 + S2 is the full portion count for that dish used in all batch and cost calculations.

Use the All / S1 / S2 toggle in the top right of the Weekly Menu, Production Planner, and Packaging to filter the view to a single session. In session-filtered mode:

  • The KPI cards update to show only that session's portions, revenue, and costs
  • The Total column shows only that session's count
  • The Production Planner recalculates batch quantities for just that session
  • The Packaging sheet hides dishes with zero portions in that session
Typical workflow: Cook S1 on Sunday, pack and deliver. Cook S2 mid-week. Use the session toggle when briefing kitchen staff so they only see what's relevant to that day's cook.
Archiving Before You Clear
Important: Before you clear the menu for a new week, always click 💾 Archive Week. This saves a permanent snapshot to Menu History. Once you clear and build a new menu, the old week cannot be recovered.

Printing the Packaging Sheet

Click the 🖨 Print button or press Ctrl+P / Cmd+P. Editing Production Values

Every cell in the production table is directly editable — click on a value and type. Changes are saved per-component in the browser. Hit ✓ Save Adjustments to persist your edits. If you later hit ⟳ Refresh from Menu, it regenerates from the menu but preserves any saved adjustments.

When to refresh: Always refresh if you've added, removed, or changed portion quantities in the Weekly Menu. The session toggle (All / S1 / S2) also triggers a recalculation — switch sessions to get per-session batch quantities. Saved overrides (notes, staff count, custom times) are preserved through a refresh — only the calculated batch quantities update.
Station Load

The Station Load card shows total estimated labour hours per station. Use this to balance your kitchen — if the Protein Station is showing 6 hours and Carb is showing 1 hour, you may need to re-assign staff or stagger start times. Stations are set on each recipe card and can be changed there.

Step-by-Step Instructions

Below the batch table, the Production Planner renders a sequential step-by-step instruction view for every component, in cook order. This is designed to be printed or displayed on a tablet in the kitchen for staff to follow without needing to access the full system.

🚚 Purchasing
Auto-generated order list · PAR warnings · Supplier grouping · Limitations
How the Order List is Generated

The Purchasing page reads every dish in the weekly menu, works out the total raw ingredient weight needed (recipe quantity per serving × total portions × scale factors), and consolidates all ingredients across all dishes. It then groups the result by supplier.

Example: Chicken Breast across 3 dishes
Tikka & Rice Bowl: 165g raw × 40 portions = 6,600g
Broccoli Protein Bowl: 165g × 1.2 scale × 15 portions = 2,970g
Total chicken to order: 9,570g (9.57kg)
PAR Level Warnings

If you've set a PAR level on an ingredient (minimum stock level in grams), the Purchasing page will highlight that row in amber if the required batch quantity falls below your PAR threshold. This means you either have enough in stock to cover the week, or you should consider ordering more than the minimum calculated.

PAR is a prompt, not a gate. The system doesn't know your actual stock levels — it only knows the threshold you set. PAR is designed to flag ingredients where running out would be critical (e.g. proteins, expensive items).
Supplier Grouping

Lines are grouped by the supplier field you set on each ingredient. If an ingredient has no supplier, it falls under "Misc". For the cleanest order list, ensure every ingredient has a supplier set. This makes it easy to call or email each supplier with a single consolidated order.

What Purchasing Doesn't Include

The order list only covers ingredients in your recipes. It does not include packaging materials, cleaning supplies, or any non-food items. Those should be managed separately. The Cost Stack is where you account for packaging costs financially, but the Purchasing page only handles food ingredients.

Session Filtering

Use the All / S1 / S2 toggle in the top right to filter the Packaging sheet to a single cooking session. In S1 or S2 mode, dishes with zero portions in that session are automatically hidden — so kitchen staff only see what they need to pack for that cook. The portion badge on each dish card updates to reflect the session count.

Tip: Print the S1 packaging sheet before Sunday's cook and the S2 packaging sheet before Wednesday's. Both can be generated from the same menu without changing anything.
Printing

Click 🖨 Print for a clean supplier-sorted order list. The search bar and navigation are hidden in print view. The printed list includes ingredient name, category, quantity in grams and kg, EP cost, estimated cost, and PAR level — everything a buyer needs for a phone or counter order.

👥 Labour Planner
Staff roster · Shift scheduling · Cost per portion · Auto-sync to Cost Stack · Archiving
Overview

The Labour tab manages your kitchen team and weekly shift plan. It calculates total hours worked, total wage cost for the week, and — most importantly — labour cost per portion, which automatically feeds into the Cost Stack. Schedule your team here each week before reviewing margins on the Cost Stack.

Staff Roster

Add every team member once. The roster is your master list — you schedule from it each week using the Shifts table. Status controls whether someone appears in the shift scheduling dropdown.

FieldPurpose
NameDisplays in the shift table and all reports
RoleHead Chef, Prep Cook, Packer, etc. Used to split the weekly cost breakdown by role type in the chart
Hourly Rate (£)Gross hourly rate. All shift costs are calculated as hours × rate. Use gross pay including any holiday accrual if relevant.
StatusActive / Part-Time appear in the shift dropdown. Casual = available for zero-hours scheduling. Inactive = hidden from all scheduling — use for staff on extended leave.
NotesFree text — food hygiene certificate expiry, emergency contacts, driving licence, allergen awareness training date, etc.
Scheduling Shifts

Click + Add Shift in the card header. Select the staff member, day of week, and start/end times. The shift modal shows a live cost preview — hours × rate — before saving. Shifts are sorted by day of week in the table. To remove a shift, click the ✕ button on its row.

Labour cost per portion = total weekly labour cost ÷ total weekly portions (pulled from the Weekly Menu). This is the figure that auto-syncs into the Cost Stack. A lean operation typically targets £1.00–£2.00 per portion; above £2.50 suggests overstaffing or low portion volume for the week.
Labour → Cost Stack Auto-Sync

You do not need to manually update the Cost Stack after scheduling shifts. Every time you open the Cost Stack, it recalculates Labour per Dish automatically from your current shifts and staff rates. The green AUTO badge on that line confirms it's live. If you need to manually override it — for example, to model a hypothetical schedule — click ✏ Override on the Labour per Dish row in Cost Stack. Click ↩ Auto to restore the sync at any time.

Labour Cost by Day Chart

The bar chart shows wage spend across each day of the week. Use this to spot imbalances — a large Saturday spend on a bulk cook day is expected, but unexpected mid-week spikes may indicate overscheduling. The cost breakdown panel shows total spend grouped by staff role.

Weekly Labour Notes

The notes field at the bottom is for anything relevant to the week's team: cover arrangements, training records, holiday approvals, or late arrival notes. Notes are saved with your data and are included in the weekly archive snapshot.

Archiving Labour

Click 💾 Archive This Week from the Labour page (or Menu, or Weekly Archive page). This captures a complete snapshot of the current staff roster, all shifts, notes, total hours, and labour cost — alongside the menu data — in a single combined weekly record. You can restore labour data from any archive entry using the 👥 Restore Labour button on the Weekly Archive page.

Auto-prompt: when you navigate away from the Labour page, the system checks whether you've already archived today. If not, and you have shifts scheduled, it will automatically open the archive checklist modal — giving you a chance to save before moving on.

Gross Margin vs Contribution Margin
MetricFormulaWhat it tells you
Contribution Margin(Sell − Food Cost) ÷ SellHow much of the sell price remains after food cost only. Ignores labour and packaging. Good for recipe-level decisions.

Target: ≥50% gross margin. Below 35% is shown in red — the business is likely not covering overheads at that level. The dashboard shows your current margin at a glance every time you open the app.
Adding and Editing Cost Items

The pre-loaded cost lines cover common meal prep costs: box, lid, sleeve, sauce tub, labour per dish, delivery, and overhead. Edit any value directly in the Edit Cost Items panel on the right — type the new amount and tab out to save. Click + Add Cost Item to add a custom line and choose its category.

Cost items are flat per-dish amounts — they don't scale with recipe size or cook yield. The box costs the same whether the dish is 600g or 900g. For cost items that do scale, model them as ingredients in the recipe instead.

Per-Dish Cost Breakdown

The right panel shows every dish on the current menu with its individual food cost, fixed cost stack, sell price, and margin. Use this to identify which dishes are dragging your margins — a dish with a 35% margin on a menu where everything else is 52% stands out immediately and may need its sell price reviewed or its recipe simplified.

Setting the Sell Price

There are two places to set sell price: a default sell price in Cost Stack settings (used for margin calculations when no dish-specific price is set), and a per-dish price in the Weekly Menu table. Per-dish prices take precedence. The Cost Stack margin summary uses the average across all current menu dishes.

Packaging Cost Advice

Packaging is one of the most underestimated costs in meal prep. A good box, lid, sleeve, cutlery sachet, and sauce tub can easily reach £0.80–£1.20 total. Price each item separately for transparency. Review supplier pricing quarterly — bulk ordering from specialist packaging suppliers typically saves 20–40% vs retail.

💾 Saving & Backups
How data is saved · The one golden rule · Backup routine · Moving to a new machine
How Data is Saved

All data saves automatically to your browser's localStorage — a private storage area tied to the specific browser and device you're using. Every action (adding, editing, deleting) saves immediately. There is no cloud sync, no account, and no server. The data lives only in the browser on the machine where you entered it.

localStorage is browser-specific. Data entered in Chrome will not appear in Firefox on the same machine, and vice versa. Data on your laptop will not sync to your phone. Pick one browser on one device as your permanent home for this system.
Dark / Light Mode

The app opens in dark mode by default. Click the 🌙 Dark mode toggle at the bottom of the left sidebar to switch to light mode. Your preference is saved in your browser and restored every time you open the file.

The One Golden Rule
Do not move or rename the HTML file once you've started entering data. localStorage keys are tied to the file path. If you move the file to a different folder or rename it, the browser treats it as a new file and shows an empty system. Your old data is not lost — move the file back to its original location and reload, and it will reappear.
Backup Routine
1
Export weekly
Click ⬇ Export in the sidebar footer. This downloads a JSON file with a date-stamped filename (e.g. 29Perf_2026-02-10.json). Save it to a cloud folder — Google Drive, Dropbox, or iCloud.
2
Archive before clearing
Before clearing the weekly menu for a new week, click 💾 Archive This Week. This saves a permanent combined menu + labour snapshot inside the system. Then export immediately so the archive is included in your backup JSON.
3
Import to restore
Click ⬆ Import in the sidebar, select your JSON file. All data loads immediately. This is also how you move your setup to a new computer or browser.
Moving to a New Computer

Export a backup JSON on the old machine. On the new machine, put the HTML file in its permanent location first — the folder you put it in is where it must stay. Open it in your browser. Click Import and select the backup JSON. All ingredients, recipes, menu, archive history, cost items, and labour data load immediately.

What the Backup Contains

The JSON export contains everything in the DB object: ingredients, recipes, menu dishes, weekly archives, production adjustments, cost items, labour staff and shifts, and all settings. It does not contain the application code — that lives in the HTML file. Keep both your HTML file and your JSON exports backed up in a cloud folder.
